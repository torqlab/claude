OIDC Trusted Publishing for semantic-release: A Complete Guide

Overview
========

When setting up semantic-release for npm package publishing, you have two main authentication options:
1. NPM_TOKEN - Traditional approach using a static token stored as a GitHub secret
2. OIDC Trusted Publishing - Modern, more secure approach using cryptographic token exchange

This document explains why OIDC Trusted Publishing is the better choice and how it works with your existing configuration.


What is OIDC Trusted Publishing?
================================

OIDC stands for OpenID Connect. Trusted Publishing is an industry standard that allows npm to verify that a package is being published from your official GitHub repository using cryptographic proofs, rather than relying on static credentials.

How it works:
- GitHub generates a time-limited OIDC token that proves the CI/CD job is running in your repository
- This token is cryptographically signed and cannot be forged
- semantic-release exchanges this token for an npm access token automatically
- The npm access token is only valid for that single publish operation
- No secrets are ever stored in GitHub, eliminating a major security risk


Why Use OIDC Instead of NPM_TOKEN?
==================================

Security Benefits:
- No secrets stored in GitHub - If your GitHub account is compromised, your npm access cannot be immediately compromised
- Token scope is limited - Each publish gets a unique, time-limited token instead of a reusable static token
- Cryptographic proof - npm can verify the package truly came from your GitHub Actions, not from a leaked token
- Audit trail - Each publish is cryptographically tied to a specific GitHub workflow run

Operational Benefits:
- No token management - No need to manually create, rotate, or revoke tokens
- Automatic setup - GitHub provides the OIDC token automatically to any workflow with id-token: write permission
- Industry standard - Used by major projects; npm officially recommends this approach
- Package provenance - npm creates a signed attestation proving the package's origin


How Your Repository is Already Configured
===========================================

Your strava-api package is already set up correctly for OIDC Trusted Publishing:

1. GitHub Actions Workflow (.github/workflows/publish.yml):
   - Has permissions: id-token: write
   - Uses GITHUB_TOKEN environment variable
   - Runs: npx semantic-release

2. Package Configuration (package.json):
   - "publishConfig": { "provenance": true }
   - This enables npm to create a signed attestation proving origin

3. semantic-release Configuration (.releaserc.json):
   - Includes @semantic-release/npm plugin
   - When id-token: write permission exists, semantic-release automatically uses OIDC

The key permission line:
    permissions:
      id-token: write

This single line enables OIDC. GitHub automatically provides the OIDC token to the workflow. semantic-release detects it and uses it instead of NPM_TOKEN.


How OIDC Token Exchange Works During Publishing
================================================

When semantic-release runs in GitHub Actions:

1. GitHub's environment provides an OIDC token (because id-token: write is set)
2. Setup Node.js workflow configures npm registry settings
3. semantic-release runs: npx semantic-release
4. The @semantic-release/npm plugin:
   - Detects the OIDC token in the environment
   - Exchanges it for an npm access token
   - Uses that token to publish to npm
5. The npm access token is time-limited and single-use
6. After publishing, the token expires automatically


Provenance: Additional Security Layer
======================================

The "provenance": true setting in package.json creates a Software Bill of Materials (SBOM) that npm signs and stores with your package. This proves:

- Exactly which GitHub Actions workflow published the package
- When it was published
- What the commit hash was
- That it hasn't been tampered with since publication

Users can verify this provenance by running:
    npm view @torqlab/strava-api --json | jq .provenance

This adds transparency and helps detect supply chain attacks.


Comparison: NPM_TOKEN vs OIDC Trusted Publishing
================================================

NPM_TOKEN Approach:
- Token stored as GitHub secret
- Reusable for any publish operation
- If leaked, attacker can publish as your package
- Requires manual token rotation
- No provenance information
- Less secure for CI/CD

OIDC Trusted Publishing:
- No token stored in GitHub secrets
- Single-use token per workflow run
- If exposed, only valid for that specific publish
- Automatic token rotation per run
- Includes package provenance attestation
- Cryptographic proof of origin
- Recommended by npm security team

Your repository chose the secure path by default.

Best Practices Going Forward
============================

1. Keep "provenance": true in package.json
2. Ensure publish workflow has permissions: id-token: write
3. Use conventional commits (feat:, fix:, chore:, etc.)
4. Never add NPM_TOKEN as a GitHub secret
5. Periodically verify provenance metadata on npm

For breaking changes that bump MAJOR version:
    feat!: breaking API change

    BREAKING CHANGE: removed legacy auth method

For testing releases locally (dry-run without publishing):
    npx semantic-release --dry-run

Or with debug output:
    DEBUG=semantic-release:* npx semantic-release --dry-run


Troubleshooting
===============

If a release fails to publish:

1. Check GitHub Actions logs for workflow errors
2. Verify id-token: write permission exists
3. Ensure package.json has valid publishConfig
4. Check npm registry connectivity
5. Review semantic-release debug logs

To enable detailed logging:
    DEBUG=semantic-release:* npx semantic-release
