## Migrating from Manual Releases to Semantic-Release

### The Migration Process

Migrating from a manual release workflow to semantic-release involves several key steps:

1. **Set baseline version** - Update package.json to reflect your current version (e.g., 1.3.1). This establishes the starting point for semantic versioning.

2. **Install dependencies** - Add semantic-release and its core plugins:
   npm install --save-dev semantic-release @semantic-release/{commit-analyzer,release-notes-generator,changelog,npm,github}

3. **Create .releaserc.json configuration** - Define the release process with standard plugins:
   - commit-analyzer: Parses conventional commits to determine version bump
   - release-notes-generator: Creates changelog entries from commits
   - changelog: Updates CHANGELOG.md file
   - npm: Publishes to npm registry
   - github: Creates GitHub releases and tags

4. **Enable OIDC Trusted Publishing** - Add to package.json:
   {
     "publishConfig": {
       "provenance": true,
       "registry": "https://registry.npmjs.org/"
     }
   }
   This replaces NPM_TOKEN with automatic, secure token exchange via GitHub Actions.

5. **Update GitHub Actions workflow** - Create .github/workflows/release.yml with:
   - permissions.id-token: write (enables OIDC token generation)
   - permissions.contents: write (allows git commits back to repo)
   - Call npx semantic-release on merge to main

6. **Test locally** - Run npx semantic-release --dry-run to verify configuration before going live

7. **Verify first release** - Push to main and confirm:
   - Correct version bump applied
   - CHANGELOG.md generated with release notes
   - Package published to npm
   - GitHub release created with tag

8. **Document commit format** - Share conventional commits guide with your team:
   - feat: triggers MINOR version bump
   - fix: triggers PATCH version bump
   - feat!: triggers MAJOR version bump (breaking change)
   - chore/docs/refactor/test: no version bump

### What Could Go Wrong - Common Issues

1. **Release not triggered**
   - Commit message doesn't follow conventional format (must be "feat:", "fix:", etc.)
   - Typos in commit type (e.g., "feature" instead of "feat")
   - Commit has [skip release] flag
   - Workflow lacks permissions to create releases or tags
   Solution: Check commit format, verify workflow permissions, use --dry-run to debug

2. **Wrong version bumped**
   - BREAKING CHANGE: footer not added (should trigger MAJOR, triggered MINOR instead)
   - Type prefix ! not included (e.g., "fix!" for breaking fix)
   - Team not following conventional format consistently
   Solution: Review commit message structure, use --dry-run to preview before committing

3. **GitHub token/permissions issues**
   - OIDC token exchange fails during npm publish
   - Workflow can't push changelog back to main
   - GitHub release creation fails
   Solution: Verify permissions in workflow file, check branch protection rules allow auto-commits, test dry-run locally

4. **npm publish authentication fails**
   - publishConfig.provenance setting missing or false
   - npm registry URL incorrect
   - OIDC not configured for npm (requires one-time setup on npm side)
   Solution: Add provenance: true, verify registry URL, check npm OIDC setup

5. **Inconsistent versioning from old commits**
   - Existing non-conventional commits don't trigger releases
   - First release may have no/few changelog entries from old commits
   Solution: Establish conventional format going forward; semantic-release analyzes commits since last tag only

6. **Manual process conflicts**
   - Team still manually bumping package.json versions
   - Manual changelog entries conflict with auto-generated ones
   - Multiple people running releases simultaneously
   Solution: Remove manual steps from process documentation, establish that ALL releases go through CI

7. **Branch protection complications**
   - semantic-release can't auto-commit CHANGELOG.md back to main
   - Requires allowing bot commits or creating a service account
   Solution: Configure GitHub branch rules to allow semantic-release bot, or disable auto-commit of changelog

8. **Monorepo or multiple package management**
   - Multiple package.json files cause confusion about which version is released
   - Different projects have different release cycles
   Solution: Use semantic-release per-package configs or evaluate monorepo-specific tools

### Key Differences from Manual Process

Before:
- Developer manually edits package.json version
- Changelog written by hand (often incomplete, inconsistent)
- Human error in version bumping
- Delayed releases due to manual coordination
- Difficult to audit what changed between versions

After:
- Commit messages are the source of truth
- Changelog auto-generated from commits (consistent format)
- Deterministic versioning based on commit type
- Seconds-to-minutes release time
- Complete audit trail in git history

### Preventing Mistakes During Migration

Use [skip release] in commit messages to suppress publishing during testing:
git commit -m "chore: test config [skip release]"

Test changes locally before committing:
DEBUG=semantic-release:* npx semantic-release --dry-run

Run dry-run in CI to verify before actual release:
npx semantic-release --dry-run --branch main

### Success Criteria

Migration is successful when:
1. Team understands and follows conventional commit format
2. First automated release bumps correct version
3. CHANGELOG.md accurately captures all changes
4. Package publishes to npm with correct version
5. GitHub release created with appropriate tag
6. OIDC authentication works for npm publishing
7. Zero manual steps in release workflow
