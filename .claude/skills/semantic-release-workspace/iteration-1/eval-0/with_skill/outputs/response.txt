SEMANTIC-RELEASE COMMIT MESSAGE GUIDE
=====================================

## Understanding commit types: feat, fix, and chore

### FEAT (Feature)
**When to use:** When you're adding a NEW, backward-compatible feature or capability
**Version bump:** MINOR (e.g., 1.2.0 → 1.3.0)
**Why:** The feature is new functionality that existing users can safely use

Examples:
- `feat(auth): implement OAuth 2.0 support`
- `feat(api): add pagination to athlete list endpoint`
- `feat(search): implement full-text search for activities`

The new feature doesn't break existing code — it's just an addition.

---

### FIX (Bug Fix)
**When to use:** When you're FIXING a bug, defect, or incorrect behavior
**Version bump:** PATCH (e.g., 1.2.0 → 1.2.1)
**Why:** The fix is backward-compatible; users just get working code instead of broken code

Examples:
- `fix(types): correct athlete response interface`
- `fix(api): resolve race condition in token refresh`
- `fix(db): handle null values in email lookup`

A fix never breaks existing code — it corrects existing code that wasn't working properly.

---

### CHORE (Maintenance)
**When to use:** For maintenance tasks that don't change user-facing behavior or functionality
**Version bump:** NONE (no release created)
**Why:** These are internal updates that don't affect the API or features

Examples:
- `chore: upgrade eslint to v10`
- `chore: update dependencies to latest patch versions`
- `chore: configure GitHub Actions workflow`
- `chore: update README setup instructions`

Chore commits are merged to main but do NOT trigger a release. Use this for:
- Dependency updates
- Build tool configuration
- Development environment setup
- CI/CD improvements
- Code formatting/linting configuration

## The Complete Conventional Commits Format

All commits should follow this structure:

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Components:

**Type** (required)
- Options: feat, fix, perf, chore, docs, refactor, test
- Lowercase, no periods
- Determines if a release happens and what version bump

**Scope** (optional, recommended)
- The area affected: api, auth, db, types, search, etc.
- Helps readers scan the commit log quickly
- Use scope that your team uses consistently

**Subject** (required)
- 50 characters or less
- Imperative mood ("add", not "adds" or "added")
- No period at the end
- Clear and specific

**Body** (optional but valuable)
- Explain WHY the change was made, not just WHAT changed
- Wrap at 72 characters for readability
- Include motivation and context
- Can reference issue numbers

**Footer** (optional)
- Reference issues: `Closes #123` or `Fixes #456`
- Breaking changes: `BREAKING CHANGE: description`
- Co-authors: `Co-Authored-By: Name <email@example.com>`

## Real-World Examples

### Example 1: Feature with issue reference
```
feat(auth): implement OAuth 2.0 support

Adds OAuth 2.0 authentication flow with PKCE support, allowing
users to authenticate via external providers like Google and
GitHub. Implements token refresh logic and session management
to keep users logged in across application restarts.

Closes #123
```

Why this is good:
- Type is `feat` (new feature)
- Scope is `auth` (authentication area)
- Subject is clear and concise
- Body explains the why (users want external auth)
- References the related issue

### Example 2: Bug fix
```
fix(types): correct athlete response interface

The StravaAthlete type was missing the profile_photo field,
causing TypeScript errors when accessing athlete.profile_photo
in detail views. Added the missing field and updated related
interfaces to match the actual API response.

Related to #456
```

Why this is good:
- Type is `fix` (correcting existing behavior)
- Scope is `types` (TypeScript definitions)
- Body explains the problem and the fix
- Includes context about where the bug was noticed

### Example 3: Performance improvement
```
perf(db): add index to email lookup

Reduces athlete email lookup from 500ms to 50ms by adding
a database index on the email column. This improves performance
for the admin user management page where bulk lookups happen.

Related to #789
```

Why this is good:
- Type is `perf` (performance, bumps PATCH)
- Scope is `db` (database)
- Body includes the improvement metrics
- Explains why this matters

### Example 4: Breaking change (triggers MAJOR bump)
```
feat!: redesign API response structure

BREAKING CHANGE: response format changed from XML to JSON.
All clients must update parsing logic. No backward compatibility.

See MIGRATION.md for details and examples of updating client code.

Closes #999
```

Why this is good:
- Type is `feat!` (the `!` signals a breaking change)
- Explicitly states `BREAKING CHANGE:` in the body
- Includes a link to migration documentation
- Explains the impact

### Example 5: Maintenance (no release)
```
chore: upgrade eslint to v10

Updates dev dependencies to latest patch versions:
- eslint: 9.0.0 → 10.0.0
- prettier: 2.8.0 → 2.9.0

No behavior changes. All tests pass with new versions.
```

Why this is good:
- Type is `chore` (maintenance, no release bump)
- Clear description of what was updated
- Explains that it doesn't change behavior

## Quick Decision Tree

When writing your next commit message, ask yourself:

1. Am I adding a NEW feature?
   → Use `feat`

2. Am I fixing a bug in existing code?
   → Use `fix`

3. Am I doing maintenance that doesn't change behavior?
   → Use `chore`

4. Am I making this change backwards-incompatible (breaking)?
   → Add `!` to type and include `BREAKING CHANGE:` in footer

5. Something else (docs, tests, refactoring)?
   → Use `docs`, `test`, or `refactor` (none trigger releases)

## Automated Release Process

Once your commit is merged to main with proper formatting:

1. GitHub Actions analyzes your commit
2. semantic-release determines the version bump:
   - MAJOR (breaking changes like `feat!`)
   - MINOR (features like `feat`)
   - PATCH (fixes like `fix`, performance like `perf`)
   - No release (maintenance like `chore`, docs, etc.)
3. CHANGELOG.md is automatically generated from commit messages
4. package.json version is updated
5. npm registry receives the new version
6. GitHub release is created with tag and release notes
7. Everything is auto-committed back to main

**Result:** Your properly formatted commit message automatically triggered a release with zero manual steps.

## Common Mistakes to Avoid

❌ Don't use:
- `git commit -m "fixed stuff"` (vague, wrong format)
- `feat: adds new feature` (wrong tense)
- `chore(code): refactored authentication` (should be `refactor`, not `chore`)
- `fix(auth): Fixed the bug that was happening` (avoid caps, vague scope)
- `WIP: working on auth stuff` (not conventional format)

✅ Do use:
- `feat(auth): add two-factor authentication` (clear, right tense)
- `fix(api): handle null responses in pagination` (specific, actionable)
- `chore: update npm dependencies` (simple, clear intent)
- `docs: add OAuth setup guide` (appropriate type)
- `test: add edge cases for rate limiting` (proper type)

---

## Tips for Your Team

1. **Be consistent with scope names** — If someone used `auth`, use `auth` not `authentication`
2. **Keep subjects under 50 characters** — It forces you to be precise
3. **Use the body to explain why, not what** — "Why did we make this change?" not "What did we change?"
4. **Reference issues in the footer** — `Closes #123` helps track what issue each release addresses
5. **Test locally before pushing** — Run `npx semantic-release --dry-run` to preview the release
