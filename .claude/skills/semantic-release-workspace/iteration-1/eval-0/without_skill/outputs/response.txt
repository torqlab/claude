Understanding Semantic-Release Commit Message Types

When using semantic-release, your commit messages determine how the version number changes and what gets included in the changelog. The tool follows the Conventional Commits specification, which uses specific prefixes to categorize changes.

THE THREE MAIN TYPES YOU'LL USE

1. FEAT (Feature) - New functionality
   - Purpose: Use this when you add a completely new feature or capability to the code
   - Version Impact: Bumps the MINOR version (e.g., 1.3.0 → 1.4.0)
   - Included in: CHANGELOG (yes, included in release notes)
   - Examples:
     * "feat(api): add rate limiting support"
     * "feat(auth): implement OAuth 2.0 flow"
     * "feat(types): add new StravaActivity interface"

   When to use: Any time you're adding something new that didn't exist before.

2. FIX (Bug Fix) - Corrections to existing functionality
   - Purpose: Use this when you're fixing a bug or correcting broken behavior
   - Version Impact: Bumps the PATCH version (e.g., 1.3.0 → 1.3.1)
   - Included in: CHANGELOG (yes, included in release notes)
   - Examples:
     * "fix(types): correct athlete response interface"
     * "fix(api): handle null values in athlete profile"
     * "fix: prevent race condition in data fetch"

   When to use: When you're making something work that was broken or incorrect.

3. CHORE (Maintenance) - Non-code changes
   - Purpose: Use this for maintenance tasks, dependency updates, tooling changes, and configuration
   - Version Impact: NO version bump (release is skipped if only chores)
   - Included in: CHANGELOG (no, not in release notes)
   - Examples:
     * "chore: upgrade eslint to v10"
     * "chore(deps): update typescript to 5.0"
     * "chore: configure pre-commit hooks"

   When to use: For anything that doesn't affect the library's functionality or API - dependencies, tooling, configuration, scripts.

OTHER TYPES YOU MIGHT ENCOUNTER

- docs: Documentation changes only (no version bump, not in changelog)
- refactor: Code restructuring without behavior changes (no version bump, not in changelog)
- test: Adding or updating tests (no version bump, not in changelog)
- perf: Performance improvements (bumps PATCH version, included in changelog)

THE COMMIT MESSAGE FORMAT

The standard format is:

```
<type>(<scope>): <subject>

<body>

<footer>
```

Example with full details:

```
feat(api): add rate limiting support

Implements exponential backoff for API rate limits to handle
high-traffic periods more gracefully.

Closes #42
```

Breakdown:
- type: feat, fix, chore, etc.
- scope (optional): affected part of the code in parentheses
- subject: short, imperative description (no period, lowercase)
- body (optional): detailed explanation
- footer (optional): reference to issues or breaking changes

SPECIAL CASE: BREAKING CHANGES

If your change breaks the existing API or behavior, trigger a MAJOR version bump (e.g., 1.3.0 → 2.0.0) by adding either:
- An exclamation mark: `feat!: breaking API change`
- A footer: Include `BREAKING CHANGE: description of what broke`

Example:
```
feat!: remove legacy authentication method

BREAKING CHANGE: The old token-based auth is no longer supported.
Use OAuth 2.0 instead.
```

DECISION TREE FOR YOUR COMMITS

Ask yourself:

1. Am I adding something completely new? → Use `feat`
2. Am I fixing a bug or correcting broken behavior? → Use `fix`
3. Am I updating dependencies, tooling, or configuration? → Use `chore`
4. Am I just documenting something? → Use `docs`
5. Am I reorganizing code without changing functionality? → Use `refactor`
6. Am I adding tests? → Use `test`
7. Am I making something faster? → Use `perf`

PRACTICAL EXAMPLES FOR YOUR WORKFLOW

Scenario 1: Adding a new API endpoint
```
feat(api): add support for athlete statistics endpoint
```
Result: Next release will be 1.4.0 (if current is 1.3.0)

Scenario 2: You find a bug in the athlete type definition
```
fix(types): add missing profile_photo field to StravaAthlete
```
Result: Next release will be 1.3.1

Scenario 3: You need to update the project's dependencies
```
chore(deps): upgrade node to 20.0.0
```
Result: No new release unless other feat/fix commits exist

Scenario 4: You're rewriting a function for clarity
```
refactor(api): simplify request handling logic
```
Result: No version bump

KEY TAKEAWAY

The tool automatically reads your commit messages and:
- Counts how many feat, fix, perf commits you made
- Determines the version bump (major, minor, or patch)
- Generates the changelog automatically
- Publishes to npm

This is why commit message discipline is important - you're essentially declaring what kind of change you made, and the tool acts on that declaration.

YOUR TEAM'S SPECIFIC SETUP

Your team's repository uses this configuration:
- Branches monitored: main
- Automatic publishing: Yes (via GitHub Actions)
- Breaking changes support: Yes (use feat! or BREAKING CHANGE footer)
- Disable a release: Add [skip release] to your commit message if needed

Now when you commit, you can use the right prefix and the rest happens automatically!
